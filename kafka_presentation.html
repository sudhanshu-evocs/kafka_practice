The Core Paradigm: What is Apache Kafka?

Apache Kafka is an open-source distributed event streaming platform designed to handle high-throughput, real-time data feeds with low latency.

  • Distributed Commit Log: Messages are appended sequentially to an immutable, write-ahead log stored on disk.
  • Publish & Subscribe: Decouples producers (data sources) from consumers (data destinations).
  • Durable Storage: Unlike traditional message queues, Kafka retains data for configurable periods, allowing history replays.
  • Pull-Based Consumer model: Consumers poll Kafka for data, controlling their own consumption rate (backpressure safety).
03

Core Architecture: Ecosystem Components

  • Brokers: Servers that receive, store, and serve messages. Together they form a clustered unit.
  • Producers: Client applications that write data (events) to Kafka brokers. They automatically select which partition to write to.
  • Consumers & Consumer Groups: Applications that read data. Grouping enables distributed scale, sharing partition consumption.
  • Zookeeper / KRaft: The coordination plane. It handles controller election, cluster membership, and metadata configurations.
  • Kafka Connect: Standardized integration framework for pulling data from databases/files or pushing to storage/indexers.
Topics & Partitions

A Topic is a logical stream category. Topics are split into Partitions which reside on different brokers to distribute load and enable parallelism.

Commit Log Durability

Writes are immediate to disk (log file) but read from memory (OS Page Cache). High performance is achieved by utilizing sequential I/O and zero-copy transfers.

04

Anatomy of a Topic: Partitions & Offsets

Each partition is an ordered, immutable sequence of records. It is the fundamental unit of scaling in Kafka.

  • Offsets: A unique, incremental 64-bit integer assigned to every message written to a partition.
  • Ordering Guarantee: Order is strictly guaranteed only within a single partition, not across the entire Topic.
  • Append Only: Data is written at the tail (highest offset) and read from sequential locations. No updates or random inserts are allowed.
  • Segments: Partitions are physically split on disk into Segment files (usually 1GB or 7 days). Old segments are deleted or compacted based on retention rules.
05

Message Structure & Partition Routing

A Kafka record is a byte array envelope consisting of metadata and payload:

  • Key: Used to determine destination partition and guarantees ordering. If null, messages are round-robined or sticky partitioned.
  • Value: The payload (serialized using JSON, Avro, Protobuf, etc.).
  • Timestamp: Auto-set by Producer (CreateTime) or Broker (LogAppendTime). Used for lifecycle and windowed operations.
  • Headers: Key-value metadata pairs. Critical for distributed tracing (correlation IDs), security tokens, and routing.
Kafka Record Byte Layout
Record Header: [Magic Byte] [Attributes] [Timestamp]
Key Length: 4 Bytes
Key: "USR-1002" (e.g. Serialized ID)
Value Length: 64 Bytes
Value: {"action": "click", "item": "laptop"}
Headers: {"traceId": "abf78-2a"}
Key Routing Math:
partition = murmurHash(key) % numPartitions
06

Consumer Groups & Partition Rebalancing

Consumers working together under the same group.id form a Consumer Group. Partitions are divided among the members.

  • Scale & Share: 1 partition is assigned to exactly 1 consumer in a group. Idle consumers occur if count exceeds partitions.
  • Rebalancing: Process of reassigning partition ownership when membership changes.
  • Triggers: Consumer crash/startup, network split, topic partition expansion.
  • Rebalance Protocols:
    • Eager (Stop-the-world): Revokes all partitions, pauses processing, then reassigns.
    • Cooperative Sticky: Iterative revocation. Keeps unchanged assignments running.
07

Replication & High Availability

Data safety in Kafka is guaranteed through replication settings across multiple physical brokers.

  • Leader Partition: Handles 100% of read and write actions. Only one broker holds the Leader for a partition.
  • Follower Partitions: Replicate data passively from the Leader. If the Leader goes offline, a Follower is elected leader.
  • In-Sync Replicas (ISR): Followers that are actively caught up with the Leader (controlled by lag configs).
  • ACKS Configurations:
    • acks=0: Fire and forget. No verification.
    • acks=1: Acknowledged by Leader only.
    • acks=all: Acked by all ISRs (Highest safety).

ZooKeeper vs. KRaft Metadata Architecture

Feature ZooKeeper Mode (Legacy) KRaft Mode (Modern / Production Ready)
Architecture Dual-system. Requires separate ZooKeeper cluster and broker consensus. Single-system. Metadata consensus is built directly into Kafka via Raft.
Partition Limit Limited to ~200,000 partitions per cluster due to ZooKeeper ZNode sync lag. Scales to millions of partitions. Minimal latency metadata log.
Controller Election Controller election occurs on ZK. Broker failures trigger re-elections. (Seconds to Minutes). Active controller is pre-elected. Failover is instantaneous (Sub-second / Milliseconds).
Operations Complex. Requires configuring, monitoring, and securing two different technologies. Simplified. Unified security protocol, port structures, configs and tooling.

Stream Processing: Kafka Streams & ksqlDB

Beyond ingestion, Kafka serves as a complete real-time computational engine.

  • Kafka Streams: A lightweight client library for Java/Scala. Runs inside your app process. No dedicated compute cluster needed.
  • Dualism of Streams and Tables:
    • KStream: Representation of an insert stream (each event is independent).
    • KTable: Representation of updates (changelog stream showing latest state per key).
  • Stateful Operations: Built-in support for joins, aggregations, and windowing using local RocksDB caches.
  • ksqlDB: Event streaming database that allows creating continuous queries using ANSI SQL syntax.
ksqlDB Real-time Aggregation
CREATE STREAM customer_clicks (
  user_id VARCHAR,
  page_id VARCHAR
) WITH (kafka_topic='clicks', value_format='json');

CREATE TABLE user_click_counts AS
SELECT user_id, count(*) AS click_count
FROM customer_clicks
WINDOW TUMBLING (SIZE 10 MINUTES)
GROUP BY user_id;

Kafka Connect: Data Integration Pipeline

Kafka Connect is a tool for scalably and reliably streaming data between Kafka and other systems.

  • No-Code Integrations: Reusable connectors configured using simple JSON. No custom programming needed.
  • Source Connectors: Ingest from external databases (CDC), queues, file directories, or web sockets.
  • Sink Connectors: Export topic records out to Elasticsearch, S3, Snowflake, PostgreSQL, etc.
  • Distributed Workers: Runs on a cluster, autoscales tasks, automatically reassigns integration jobs on failures.

from kafka import KafkaProducer import json # Initialize Producer matching client's local app/producer.py setup producer = KafkaProducer( bootstrap_servers="localhost:9092", value_serializer=lambda v: json.dumps(v).encode("utf-8") ) event = {"incident_id": "INC-1001", "severity": "High", "site": "Delhi DC"} # Send is asynchronous by default future = producer.send("ehs-incidents", value=event) # Wait for response acknowledgement (optional - turns call into synchronous) record_metadata = future.get(timeout=10) print(f"Sent to topic {record_metadata.topic} partition {record_metadata.partition} offset {record_metadata.offset}") producer.flush() # Force sending all buffered records producer.close()
from confluent_kafka import Producer conf = { 'bootstrap.servers': "localhost:9092" } producer = Producer(conf) def delivery_report(err, msg): """ Callback method invoked when message is acknowledged or failed. """ if err is not None: print(f"Message delivery failed: {err}") else: print(f"Message delivered to {msg.topic()} [{msg.partition()}] at offset {msg.offset()}") producer.produce( topic='ehs-incidents', value='{"severity": "High", "site": "Mumbai"}', callback=delivery_report ) # Serve callbacks from buffer queue producer.poll(1) producer.flush()
from confluent_kafka import Producer conf = { 'bootstrap.servers': "localhost:9092", # Idempotent configurations 'enable.idempotence': True, # Ensures exactly-once delivery from producer to broker 'acks': 'all', # Wait for all ISR to acknowledge 'max.in.flight.requests.per.connection': 5, # Keeps order even on retries 'retries': 5, # Retry writing on temporary broker failure # High-throughput batch tuning 'linger.ms': 20, # Wait up to 20ms to batch messages 'compression.type': 'snappy' # Compress data before sending } producer = Producer(conf) producer.produce('ehs-incidents', value='payload') producer.flush()

Case Study: EHS Incident Event Pipeline

Our workspace implements a real-world multi-consumer pipeline using the ehs-incidents topic.

  • Single Source of Truth: producer.py streams incident events (ID, Severity, Site) into Kafka.
  • Parallel Decoupled Pipelines:
    • alert-group (alert_consumer.py): Filters "High" severity issues instantly to trigger operations team notifications.
    • db-group (db_consumer.py): Archives all events into persistent storage for auditing.
    • ml-group (ml_consumer.py): Feeds events directly into ML models for predicting escalation probabilities.
  • Fault Isolation: Slow DB transactions do not delay critical emergency alarms.

Workspace Code: EHS ML Risk Consumer

app/ml_consumer.py (Production Ready)
from kafka import KafkaConsumer import json class IncidentRiskModel: def __init__(self): self.site_risk_weights = {"Delhi DC": 0.4, "Mumbai DC": 0.35, "Bangalore DC": 0.25} self.severity_multipliers = {"High": 2.5, "Medium": 1.5, "Low": 0.8} def predict_escalation_probability(self, site, severity): base_weight = self.site_risk_weights.get(site, 0.3) multiplier = self.severity_multipliers.get(severity, 1.0) return min(max(base_weight * multiplier, 0.0), 1.0) # Consume events from the EHS-Incidents topic consumer = KafkaConsumer( "ehs-incidents", bootstrap_servers="localhost:9092", auto_offset_reset="earliest", group_id="ml-group", # Independent consumer group ID value_deserializer=lambda x: json.loads(x.decode("utf-8")) ) model = IncidentRiskModel() print("ML Risk Consumer Listening...") for message in consumer: incident = message.value prob = model.predict_escalation_probability(incident.get("site"), incident.get("severity")) print(f"[{incident.get('incident_id')}] Escalation Probability: {prob:.2f}")

Wrap Up

Summary & Discussion

Kafka is a critical backbone for modern distributed systems. Decoupling storage and compute enables scalable event streaming.

Key Takeaways
  • Always use partition keys to enforce message routing order.
  • Design consumer poll limits carefully to prevent group rebalances.
  • Transition from legacy ZooKeeper to KRaft for production environments.
Q&A Topics
  • What is our current partition key routing strategy?
  • How will we handle schema migrations (Avro vs JSON)?
  • Are our consumer commit intervals properly configured for backpressure?
